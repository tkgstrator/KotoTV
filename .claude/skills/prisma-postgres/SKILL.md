# Prisma + Postgres

Schema lives in `packages/server/prisma/schema.prisma`. Dev DB runs in the devcontainer's `postgres` service and can be inspected with pgadmin. Production DB runs alongside the app via `compose.yaml`.

## Setup (Phase 0)

```sh
cd packages/server
bun add @prisma/client
bun add -D prisma
bunx prisma init --datasource-provider postgresql
```

`prisma/schema.prisma` baseline:

```prisma
generator client {
  provider = "prisma-client-js"
  output   = "../src/generated/prisma"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

> The `generated/prisma` directory is git-ignored and regenerated by `prisma generate`.

## Migration workflow

**Always go through Prisma Migrate — never raw DDL, never `db push` on committed branches.**

```sh
# 1. edit schema.prisma
# 2. generate SQL + apply to dev DB:
bunx prisma migrate dev --name add_channels
# 3. commit: prisma/schema.prisma + prisma/migrations/<timestamp>_add_channels/
```

- `migrate dev` auto-runs `prisma generate`.
- CI / production: `bunx prisma migrate deploy` (no interactive prompts).
- Reset dev DB: `bunx prisma migrate reset` (drops + re-applies + seeds).

> **Any schema change requires all three: `migrate dev` + `prisma generate` + restart `bun dev`**. `bun --hot` does not reload the generated Prisma client cleanly, so a live dev server will keep using the stale client and 500 on new tables / columns until restarted. Always ask the user to Ctrl+C and `bun dev` again after a Prisma migration.

## Models for this project

Initial models (grow during Phase 4):

```prisma
model Channel {
  id         String   @id                // Mirakc service id string
  name       String
  type       String                      // 'GR' | 'BS' | 'CS'
  serviceId  Int      @map("service_id")
  networkId  Int      @map("network_id")
  createdAt  DateTime @default(now())    @map("created_at")

  programs   Program[]
  schedules  RecordingSchedule[]

  @@map("channels")
}

model Program {
  id         String   @id
  channelId  String   @map("channel_id")
  startAt    DateTime @map("start_at")
  endAt      DateTime @map("end_at")
  title      String
  description String?
  genre      String?

  channel    Channel  @relation(fields: [channelId], references: [id], onDelete: Cascade)

  @@index([channelId, startAt])
  @@map("programs")
}

model RecordingSchedule {
  id         String   @id @default(cuid())
  channelId  String   @map("channel_id")
  programId  String?  @map("program_id")
  startAt    DateTime @map("start_at")
  endAt      DateTime @map("end_at")
  title      String
  createdAt  DateTime @default(now()) @map("created_at")

  channel    Channel  @relation(fields: [channelId], references: [id])
  recording  Recording?

  @@map("recording_schedules")
}

model Recording {
  id          String   @id @default(cuid())
  scheduleId  String   @unique @map("schedule_id")
  filePath    String   @map("file_path")
  sizeBytes   BigInt   @map("size_bytes")
  durationSec Int      @map("duration_sec")
  createdAt   DateTime @default(now()) @map("created_at")

  schedule    RecordingSchedule @relation(fields: [scheduleId], references: [id], onDelete: Cascade)

  @@map("recordings")
}
```

## Client instantiation (Bun + Hono)

Single instance per process — **do not** construct `PrismaClient` per request.

```ts
// packages/server/src/lib/prisma.ts
import { PrismaClient } from '../generated/prisma'
import { config } from './config'

export const prisma = new PrismaClient({
  log: config.NODE_ENV === 'development' ? ['query', 'warn', 'error'] : ['warn', 'error'],
})

process.on('SIGTERM', async () => {
  await prisma.$disconnect()
})
```

Usage:

```ts
import { prisma } from '../lib/prisma'

export const channelsService = {
  list: (type?: ChannelType) => prisma.channel.findMany({
    where: type ? { type } : undefined,
    orderBy: { serviceId: 'asc' },
  }),
}
```

## Connection pooling

Postgres + Prisma: the default pool is 2 × CPU cores, which is fine for this scale. If you ever front Postgres with PgBouncer, set `?pgbouncer=true` and **use transaction-mode pooling** (Prisma already accommodates).

## Inspecting data

- pgadmin: `http://localhost:8080` (already wired) — register the `postgres` server once: Host=`postgres`, Port=`5432`, User=`postgres`, Password=`password`.
- `bunx prisma studio` — alternative, opens its own UI at `:5555`.

## `$queryRaw` and tagged templates

Use only when Prisma's query API can't express it. Always tagged-template form:

```ts
await prisma.$queryRaw<Row[]>`SELECT * FROM channels WHERE type = ${type}`
```

Never interpolate untrusted strings inside a raw SQL literal.

## Common pitfalls

- Running `prisma db push` on a branch that already has migrations — it creates drift. Always use `migrate dev`.
- Forgetting to commit `prisma/migrations/**` — other devs will regenerate and diverge.
- Instantiating `new PrismaClient()` in a route handler → connection leak and eventual exhaustion.
- Using `Json` or enum types without ensuring the Postgres extension / version supports them. Postgres 17 supports both natively.
